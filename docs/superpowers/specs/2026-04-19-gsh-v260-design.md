# GSH v2.6.0 Design Spec

## Overview

Daily delinquent summary email sent to `admin@stunna.xyz` at 4:10 AM Eastern every day. Each delinquent subscriber gets an HTML card with their account info, days overdue, amount due, and two action links: one to the website subscriber detail page, one to a Telegram deep link that triggers the existing two-step payment recording flow.

---

## Feature 1 — Daily Summary Email

### Scheduler

New file `web/backend/daily_summary.py` — asyncio coroutine `run_daily_summary()`:

- On startup, calculates seconds until next 4:10 AM US/Eastern using Python 3.11 built-in `zoneinfo`
- Sleeps until that time, fires the email, then loops with a fixed 24-hour sleep
- If the scheduled time has already passed today (e.g. container restarted at 5 AM), sleeps until 4:10 AM the following day
- If no delinquent subscribers found, still sends email with message "No delinquent accounts — all clear today."
- Errors are caught and logged; a failed send does not crash the loop

Added to `main.py` lifespan alongside `run_heartbeat`:
```python
summary_task = asyncio.create_task(run_daily_summary())
```
Cancelled cleanly on shutdown alongside the heartbeat task.

### Email content

One HTML card per delinquent subscriber, sorted by days overdue descending (most overdue first). Each card contains:

- **Account ID** (e.g. `dtv.020`)
- **Name** (username)
- **Days Overdue** — `abs(days_until_due)` days (since `days_until_due` is negative for overdue)
- **Amount Due** — `$price.2f`
- **[View on Website]** button → `https://gsh.stunna.xyz/subscribers/{acc_id}` (requires login)
- **[Record via Telegram]** button → `https://t.me/{BOT_USERNAME}?start=pay_{acc_id}`

Email header shows total delinquent count and today's date.

Email is sent via `notify_email_html()` in `notify.py`:
- **From:** `billing@guardianstreams.com` (via Gmail SMTP configured in `.env`)
- **To:** `admin@stunna.xyz` (hardcoded — this is an admin-only internal email)
- **Subject:** `GSH Daily Summary — {N} Delinquent Account(s) — {date}`
- Uses existing `EMAIL_*` env vars already present in `.env`

## Feature 2 — `notify_email_html()` in `notify.py`

New function added to `notify.py`:

```python
def notify_email_html(subject: str, html_body: str, to: str) -> bool:
```

- Reads `EMAIL_*` config from `CONFIG["NOTIFICATIONS"]["EMAIL"]`
- Returns `False` silently if `enabled` is False or credentials are missing
- Sends via `smtplib.SMTP` with STARTTLS on port 587
- `MIMEMultipart("alternative")` with plain-text fallback: "Open this email in an HTML-capable client to view the summary."
- Returns `True` on success, logs warning and returns `False` on any exception

## Feature 3 — Telegram `/start pay_*` handler + bot username cache

### Bot username

On module load in `telegram.py`, fetch the bot username once via `getMe`:

```python
_bot_username: str = ""

def _fetch_bot_username() -> str:
    token = CONFIG["NOTIFICATIONS"]["TELEGRAM"].get("bot_token", "")
    if not token:
        return ""
    try:
        r = requests.get(f"https://api.telegram.org/bot{token}/getMe", timeout=10)
        if r.status_code == 200 and r.json().get("ok"):
            return r.json()["result"]["username"]
    except Exception:
        pass
    return ""

_bot_username = _fetch_bot_username()
```

`_bot_username` is used by `daily_summary.py` to build deep links:
```python
from routers.telegram import _bot_username
f"https://t.me/{_bot_username}?start=pay_{acc_id}"
```

If `_bot_username` is empty (Telegram not configured), the Telegram button is omitted from the email card.

### `/start pay_*` webhook handler

In the existing `telegram_webhook()` function, add a check inside the `"message"` block before the `_pending` check:

```python
if text.startswith("/start pay_"):
    acc_id = text[len("/start pay_"):]
    sub = get_subscriber_by_id(acc_id)
    if sub:
        _send_notice_message(chat_id, acc_id, sub)
    else:
        _send_message(chat_id, f"Account {acc_id} not found.")
    return {"ok": True}
```

`_send_notice_message(chat_id, acc_id, sub)` sends the due notice with the inline 💳 Record Payment button directly to `chat_id` (the user who tapped the deep link) — same message content as `_send_notice()` in `subscribers.py`, but targets a specific `chat_id` instead of the configured admin `chat_id`.

Note: `_send_notice()` in `subscribers.py` sends to `cfg['chat_id']` (the configured admin chat). The new `_send_notice_message()` in `telegram.py` takes an explicit `chat_id` parameter so it works when triggered from a deep link.
