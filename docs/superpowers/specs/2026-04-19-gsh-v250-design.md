## Overview

Four improvements to the GSH web app across UI polish, UX state persistence, and a new Telegram-based payment recording flow.

## Feature 2 — Status Filter Persistence (sessionStorage)

### What

The "All Statuses" dropdown on the Subscribers page (`web/frontend/src/pages/Subscribers.tsx`) persists its value in `sessionStorage` so it survives page navigation within the same browser session but resets when the tab/window is closed.

### Behaviour

- On mount: read `sessionStorage.getItem('gsh_status_filter')` — if present, use as initial value; otherwise default to `''` (All Statuses)
- On change: call `sessionStorage.setItem('gsh_status_filter', value)` alongside existing state setter
- No backend change required

## Feature 3 — Telegram Payment via Webhook

### Overview

A two-step Telegram flow: a "Send Due Notice" button in the subscriber detail page sends a Telegram message with an inline "💳 Record Payment" button. Tapping that button starts a reply flow where the bot shows the current price — the user replies with an amount (or `ok` to confirm) and the payment is recorded automatically.

### Backend

#### New file `web/backend/routers/telegram.py`

Registered in `main.py` under prefix `/api/telegram`. Contains only the webhook routes.

#### Modified file `web/backend/routers/subscribers.py`

Single and bulk due-notice routes added here (prefix `/api/subscribers`) so URLs stay consistent with the rest of the subscribers API. Route ordering: `bulk/send-due-notices` must be declared before `{acc_id}/send-due-notice` to avoid path conflicts.

#### Route: `POST /api/subscribers/{acc_id}/send-due-notice` (in `subscribers.py`)

- Auth: `verify_api_key` (called from the web frontend)
- Looks up the subscriber from DB (username, price, due_date, days_until_due)
- Sends a Telegram message via `requests.post` to `sendMessage` with an `InlineKeyboardMarkup`:
  ```
  📅 Payment Due Soon
  Account: {acc_id}
  Name: {username}
  Amount: ${price:.2f}
  Due in: {days_until_due} days
  ```
  Inline button label: `💳 Record Payment`  
  Callback data: `pay:{acc_id}:{price:.2f}`
- Returns `{"message": "Due notice sent for {acc_id}"}` or raises 400/502

#### Route: `POST /api/telegram/webhook` (in `telegram.py`)

- **No API key auth** — Telegram calls this endpoint directly
- Accepts the raw Telegram `Update` JSON body
- Handles two cases:

**Case A — Callback query** (user tapped the inline button):
1. Parse `callback_data` — must match pattern `pay:{acc_id}:{price}`
2. Store in module-level `_pending: dict[int, dict]` keyed by `chat_id`: `{"acc_id": acc_id, "price": float(price)}`
3. Answer the callback query (required by Telegram) via `answerCallbackQuery`
4. Send reply message:
   ```
   💳 Recording payment for {username} ({acc_id})
   Current price: ${price:.2f}
   
   Reply with the amount, or send 'ok' to confirm ${price:.2f}.
   ```

**Case B — Text message** (user replied with amount or `ok`):
1. Check if `chat_id` is in `_pending`
2. If not — ignore silently (not a payment flow)
3. If yes — parse text:
   - `ok` (case-insensitive) → use stored price
   - Numeric string → parse as float amount
   - Anything else → reply "Please send a number or 'ok'" and return
4. Call `add_payment(acc_id=..., amount=..., status="paid", advance_days=30)`
5. Look up new due_date from DB
6. Reply:
   ```
   ✅ Payment of ${amount:.2f} recorded for {acc_id}.
   New due date: {new_due_date}
   ```
7. Remove `chat_id` from `_pending`
8. Return HTTP 200 (Telegram requires 200 even on handled errors)

#### Route: `GET /api/telegram/webhook/set` (in `telegram.py`)

- Auth: `verify_api_key`
- Calls Telegram `setWebhook` with URL `https://gsh.stunna.xyz/api/telegram/webhook`
- Returns Telegram's response JSON
- Called once manually after deploy

#### State

`_pending` is a module-level dict — in-memory, no DB. If the container restarts mid-flow, the user taps the button again. No cleanup needed for abandoned flows (dict stays small in practice).

### Frontend — `SubscriberDetail.tsx`

- Add `📨 Send Due Notice` button with `variant="teal"` to the Account Info card header
- Mutation: `POST /api/subscribers/{acc_id}/send-due-notice`
- On success: toast "Due notice sent via Telegram"
- On error: toast with error message

### API — `web/frontend/src/lib/api.ts`

Add: `export const sendDueNotice = (accId: string) => api.post(`/subscribers/${accId}/send-due-notice`).then(r => r.data)`

## Feature 4 — Bulk "Send Due Notice" for Delinquent Subscribers

### Backend

#### Route: `POST /api/subscribers/bulk/send-due-notices` (in `subscribers.py`)

- Auth: `verify_api_key`
- Optional body: `{"account_ids": ["ACC-001", ...]}` — if omitted, queries all active subscribers with `status = 'delinquent'`
- For each subscriber, calls the same Telegram send logic as the single due-notice route
- Returns `{"sent": N, "failed": N, "message": "Sent N due notices"}`
- Uses `BackgroundTasks` — fires and forgets, returns immediately with the count

**Note:** This route must be registered before `/{acc_id}` routes in the router to avoid path conflicts.

### Dashboard — `web/frontend/src/pages/Dashboard.tsx`

New card "Delinquent Notices":
- Header shows count of delinquent subscribers (from existing subscribers query, filtered by status)
- Single button: `📨 Send Due Notices to All Delinquent`
- On click: POST to `/api/subscribers/bulk/send-due-notices` (no body — targets all delinquent)
- On success: toast showing "Sent N due notices"

### Subscribers page — `web/frontend/src/pages/Subscribers.tsx`

When the status filter dropdown is set to `delinquent`:
- A `📨 Send Due Notices` button appears inline in the filter toolbar (to the right of the dropdowns)
- On click: POST to `/api/subscribers/bulk/send-due-notices` with `{"account_ids": [...ids of currently visible subscribers]}`
- On success: toast showing result

### API — `web/frontend/src/lib/api.ts`

Add:
```typescript
export interface BulkNoticeResult {
  sent: number
  failed: number
  message: string
}
export const bulkSendDueNotices = (accountIds?: string[]) =>
  api.post<BulkNoticeResult>('/subscribers/bulk/send-due-notices', accountIds ? { account_ids: accountIds } : {}).then(r => r.data)
```
